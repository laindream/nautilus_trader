# 文档风格指南

本指南概述了为NautilusTrader编写文档的风格约定和最佳实践。

## 一般原则

- 我们偏爱简单而非复杂，少即是多。
- 我们偏爱简洁而又可读的散文和文档。
- 我们重视约定、风格、模式等方面的标准化。
- 文档应该对不同技术背景的用户都是可访问的。

## 语言和语调

- 尽可能使用主动语态（"配置适配器"vs"适配器应该被配置"）。
- 描述当前功能时使用现在时。
- 只对计划中的功能使用将来时。
- 避免不必要的行话；在首次使用时定义技术术语。
- 直接而简洁；避免"基本上"、"简单地"、"只是"等填充词。

## Markdown表格

### 列对齐和间距

- 根据每列中最宽内容所需的空间使用对称的列宽。
- 垂直对齐列分隔符（`|`）以提高可读性。
- 在单元格内容周围使用一致的间距。

### 注释和描述

- 所有注释和描述都应该有结束句号。
- 保持注释简洁但信息丰富。
- 使用句子大小写（只大写首字母和专有名词）。

### 示例

```markdown
| 订单类型               | 现货 | 杠杆   | USDT期货     | 币本位期货   | 注释                    |
|------------------------|------|--------|--------------|--------------|-------------------------|
| `MARKET`               | ✓    | ✓      | ✓            | ✓            |                         |
| `STOP_MARKET`          | -    | ✓      | ✓            | ✓            | 现货不支持。            |
| `MARKET_IF_TOUCHED`    | -    | -      | ✓            | ✓            | 仅限期货。              |
```

### 支持指示符

- 使用`✓`表示支持的功能。
- 使用`-`表示不支持的功能（不是`✗`或其他符号）。
- 为不支持的功能添加注释时，用斜体强调：`*不支持*`。
- 当不需要内容时保留单元格为空。

## 代码引用

- 对内联代码、方法名、类名和配置选项使用反引号。
- 对多行示例使用代码块。
- 引用函数或代码位置时，包含模式`file_path:line_number`以便于导航。

## 标题

我们遵循优先考虑可读性和可访问性的现代文档约定：

- 对主页面标题（仅# 1级）使用标题大小写。
- 对所有子标题（## 2级及以下）使用句子大小写。
- 无论标题级别如何，始终大写专有名词（产品名称、技术、公司、缩写）。
- 确保正确的标题层次结构（不要跳过级别）。

此约定与包括Google开发者文档、Microsoft Docs和Anthropic文档在内的主要技术公司使用的行业标准保持一致。
它提高了可读性，减少了认知负荷，对国际用户和屏幕阅读器更加友好。

### 示例

```markdown
# NautilusTrader开发者指南

## Python入门
## 使用Binance适配器
## REST API实现
## WebSocket数据流
## pytest测试
```

## 列表

- 对无序列表项目符号使用连字符（`-`）；避免使用`*`或`+`以保持整个项目的Markdown风格一致。
- 只有在顺序重要时才使用编号列表。
- 对嵌套列表保持一致的缩进。
- 当列表项是完整句子时以句号结尾。

## 链接和引用

- 使用描述性链接文本（避免"点击这里"或"此链接"）。
- 适当时引用外部文档。
- 确保所有内部链接都是相对的且准确的。

## 技术术语

- 基于Nautilus领域模型建立能力矩阵，而不是交易所特定的术语。
- 必要时在括号或注释中提及交易所特定术语以便澄清。
- 在整个文档中使用一致的术语。

## 示例和代码样本

- 提供实用的、可工作的示例。
- 包含必要的导入和上下文。
- 使用现实的变量名和值。
- 添加注释来解释示例中不明显的部分。

## 警告和注释

- 对重要信息使用适当的提示块：
  - `:::note`用于一般信息。
  - `:::warning`用于重要注意事项。
  - `:::tip`用于有用的建议。

## 行长度和换行

- 换行不超过约100-120个字符以提高可读性和差异审查。
- 在自然点（逗号、连词或短语后）断开长句。
- 尽可能避免孤立的词换行。
- 必要时代码块和URL可以超出行限制。

## API文档

- 清楚地记录参数和返回类型。
- 为复杂的API包含使用示例。
- 解释任何副作用或重要行为。
- 保持参数描述简洁但完整。
